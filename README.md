# Groovy

## Overview

This project is a full-stack music recommendation system that models songs and user preferences as points in a 768-dimensional vector space. Recommendations are generated by finding songs closest to a user’s evolving taste vector, allowing for more personalized and dynamic results than traditional genre-based systems.

The application integrates with the Spotify Web API for music data, stores vector embeddings in Pinecone, caches frequently accessed data with Redis, and persists user data in PostgreSQL. A React frontend provides the user interface.

---

## Architecture

The system is composed of the following components:

- **Frontend:** React application for user interaction
- **Backend:** Node.js with Express serving API endpoints
- **Database:** PostgreSQL for persistent user and play history data
- **Cache:** Redis for fast retrieval of recommendation data and taste vectors
- **Vector Store:** Pinecone for approximate nearest neighbor (ANN) search
- **External API:** Spotify Web API for track metadata

### Data Flow

1. A user plays a song
2. The backend updates the user's taste vector based on the song vector
3. The updated taste vector is cached in Redis
4. When recommendations are requested:
   - If cached, Redis returns results immediately
   - If not cached, Pinecone performs a vector similarity search
5. Results are enriched with metadata from PostgreSQL and/or Spotify
6. The final recommendation list is returned to the frontend

## Bottlenecks

### 1. Pinecone Queries on Cold Cache
When the Redis cache is cold, recommendation requests must query Pinecone directly. While Pinecone handles ANN queries efficiently, high volumes of cache misses can increase load.

**Mitigation:**
- Cache-aside pattern
- 10-minute TTL on taste vectors

---

### 2. Spotify API Rate Limits
All track metadata is fetched from the Spotify Web API, which enforces rate limits—especially in development mode.

**Mitigation:**
- Aggressive caching of track data after initial fetch
- Avoid duplicate API calls for the same track

---

### 3. PostgreSQL Write Contention
Each song play results in a write to the `play_history` table. At scale, this becomes a bottleneck.

**Mitigation:**
- Connection pooling
- Potential future improvements:
  - Batched inserts
  - Write queue

## Key Learnings

### 1. Adapting to API Changes
Spotify deprecated audio features for new applications, requiring a shift to proxy features such as:
- Popularity
- Release year
- Artist ID
- Explicit flag

This required rethinking similarity and vector construction.

---

### 2. Cache Invalidation Strategy
Balancing freshness and performance is non-trivial:
- Recommendations can tolerate slight staleness
- Playlist and user-facing data must remain fresh

---

### 3. Optimistic Locking
Using a version column for concurrency control proved to be:
- Simple
- Efficient
- Easier than transactional locking

---

### 4. Vector-Based Recommendations
Vector similarity produces more nuanced and personalized recommendations compared to:
- Genre matching
- Tag-based systems
